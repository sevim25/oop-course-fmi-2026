# 08. Шаблони. Шаблонни функции и класове. Variadic templates. Concepts

Шаблоните в C++ са механизъм, чрез който можем да дефинираме функционалност или клас (структура от данни) с **общо предназначение спрямо типа**. 
Тоест дефинираме общо поведение, без предварително да фиксираме конкретния тип, с който ще работим.

Шаблоните са средство за писане на общ и преизползваем код. Те позволяват една и съща логика да се прилага върху различни типове, без да се дублира имплементацията.

Например вместо да пишем отделна функция за `int`, отделна за `double` и отделна за `std::string`, можем да напишем една шаблонна функция, а компилаторът да генерира нужната конкретна версия според типа, с който тя се използва.

Същата идея важи и за класовете. Вместо да имаме различни класове със сходна логика, можем да опишем един шаблонен клас, който работи с параметризирани типове.

Шаблоните реализират **параметричен полиморфизъм**. Това означава, че един и същ код може да работи с различни типове, защото типът е подаден като параметър.

Често шаблоните се разглеждат и като форма на **compile-time polymorphism**, защото конкретната версия на кода се създава по време на компилация.

Добре е да се прави следното уточнение:

- **параметричен полиморфизъм** е по-точното понятие, когато говорим за шаблони;
- **compile-time polymorphism** е по-широк термин и често включва и други механизми, например function overloading.

---

## Как работят шаблоните

Шаблонът сам по себе си не е окончателна функция или окончателен клас. Той е по-скоро **описание** или **схема**, от която компилаторът генерира конкретен код.

Когато използваме шаблона с даден тип, компилаторът извършва **инстанциране** на шаблона — създава конкретна версия, в която параметърът на шаблона е заместен с реален тип.

Например, ако имаме шаблонна функция `sum<T>`, то при използване с `int` компилаторът генерира (преписва) версия на същата като замества типа с `int`, а при използване с `double` — версия за `double`.

Точно затова шаблоните са удобни: логиката се описва веднъж, а конкретните реализации се създават автоматично.

## Шаблонни функции

Шаблонната функция е функция, чийто тип или типове не са фиксирани предварително, а се задават чрез шаблонни параметри.

```cpp
#include <iostream>

template <typename T>
T sum(const T& a, const T& b) {
    return a + b;
}

int main() {
    int a = 4;
    int b = 9;
    std::cout << sum(a, b) << '\n';

    double c = 3.14;
    double d = 4.5;
    std::cout << sum(c, d) << '\n';
}
```

Шаблонът е дефиниран така:

```cpp
template <typename T>
T sum(const T& a, const T& b)
```

Тук `T` е **шаблонен параметър**. На мястото на `T` по-късно ще бъде поставен конкретен тип.

При извикване на:

```cpp
sum(a, b)
```

компилаторът анализира типовете на аргументите и извежда стойността на `T`:

- в първото извикване `T = int`;
- във второто извикване `T = double`.

Тоест компилаторът ще генерира дефиниции на тази функция, като замести `T` съответно с `int` и с `double`.

### Явно задаване на шаблонния аргумент

Понякога можем сами да укажем с какъв тип искаме да се инстанцира шаблонът:

```cpp
std::cout << sum<int>(a, b) << '\n';
std::cout << sum<double>(c, d) << '\n';
```

В много случаи това не е необходимо, защото компилаторът може сам да изведе типа.

---

## `typename` и `class`

В декларацията на шаблона често се използва:

```cpp
template <typename T>
```

но може да се срещне и:

```cpp
template <class T>
```

В този контекст двете форми са еквивалентни. И двете означават, че `T` е параметър, който трябва да бъде тип.

В съвременния стил по-често се предпочита `typename`, защото по-ясно показва, че става дума за тип.

## Шаблонни класове

Шаблонният клас е клас, който използва шаблонни параметри, вместо да работи с един конкретен тип.

### Основен пример

```cpp
#include <iostream>
#include <string>

template <typename T>
class Box {
private:
    T data;

public:
    Box(const T& data) : data(data) {}

    const T& get() const {
        return data;
    }
};

int main() {
    Box<int> x(42);
    Box<std::string> y("hello");

    std::cout << x.get() << '\n';
    std::cout << y.get() << '\n';
}
```

### Какво е важно да се разбере

`Box<T>` е шаблон на клас. Той не е конкретен клас сам по себе си.

Конкретни класове са например:

- `Box<int>`
- `Box<std::string>`

Може да се мисли за тях като за различни класове, генерирани по една и съща схема.

---

## Инстанциране на шаблон

Под **инстанциране** се разбира процесът, при който компилаторът създава конкретна версия на шаблон за определен тип.

Например:

```cpp
Box<int> a(10);
Box<double> b(3.14);
```

води до инстанциране на шаблона `Box<T>` за:

- `T = int`
- `T = double`

Този момент е съществен, защото именно тогава компилаторът проверява дали използваният тип удовлетворява изискванията на шаблона.

---

## Необходими функции и необходими операции

При шаблоните често говорим за **необходими функции** или **необходими операции**.

Това са функции, методи или оператори, които самият шаблон използва, но не дефинира. Следователно типът, с който се инстанцира шаблонът, трябва да ги предоставя.

### Пример с необходим оператор

```cpp
template <typename T>
T sum(const T& a, const T& b) {
    return a + b;
}
```

Тук шаблонът изисква за типа `T` да съществува `operator+`.

Например следната функция:

```cpp
template <typename T>
T maxValue(const T& a, const T& b) {
    return (a < b) ? b : a;
}
```

ще работи само за типове, за които е дефиниран операторът `<`.

Следователно шаблоните са общи, но не безусловно. Те поставят изисквания към типа, макар тези изисквания често да не са изписани формално.

### Пример с необходим метод

```cpp
template <typename T>
void printObject(T& obj) {
    obj.print();
}
```

Тук шаблонът очаква обектът от тип `T` да има метод `print()`.

### Кога се открива грешката

Това е важен момент: компилаторът обикновено не дава грешка още при самата дефиниция на шаблона. Грешката се появява при **инстанциране**, когато шаблонът бъде използван с конкретен тип, който не покрива изискванията.

### Пример

```cpp
#include <iostream>

struct Test {
    void func() {
        std::cout << "func()\n";
    }
};

template <typename T>
void f1(T& obj) {
    obj.func();
}

template <typename T>
void f2(T& obj) {
    obj.func1();
}

int main() {
    Test t;
    f1(t);   // OK
    f2(t);   // compile error
}
```

Причината е, че `Test` има `func()`, но няма `func1()`.

Този пример показва, че шаблонът може да бъде синтактично коректен, но да не може да се използва с конкретен тип.

Ако функцията `f2(t)` не бъде извикана, то програмата ще се изпълни без грешки.

## Шаблонна специализация

Понякога искаме шаблонът да има специална реализация за конкретен тип.
Причината не винаги е различно поведение — много често специализираме шаблон, защото за даден тип можем да напишем по-ефикасна имплементация.

**Пример**: сортиране на масив

```cpp
#include <iostream>
#include <utility>

template <typename T>
void sort(T* arr, size_t n) {
    // merge sort
}

template <>
void sort<unsigned char>(unsigned char* arr, size_t n) {
    // counting sort
    size_t count[256] = { 0 };

    for (size_t i = 0; i < n; i++) {
        count[arr[i]]++;
    }

    size_t index = 0;
    for (size_t value = 0; value < 256; value++) {
        while (count[value] > 0) {
            arr[index++] = static_cast<unsigned char>(value);
            count[value]--;
        }
    }
}

int main() {
    int arr1[] = { 5, 1, 4, 2, 3 };
    sort(arr1, 5);

    for (int x : arr1) {
        std::cout << x << ' ';
    }
    std::cout << '\n';

    unsigned char arr2[] = { 5, 1, 255, 2, 1, 0, 255 };
    sort(arr2, 7);

    for (unsigned char x : arr2) {
        std::cout << static_cast<int>(x) << ' ';
    }
    std::cout << '\n';
}
```

Имаме обща версия на `sort`, която работи за произволен тип T, стига да може да сравняваме елементите с `<`.

Тази версия например може да използва алгоритъма за сортиране _Merge sort_.

За `unsigned char` обаче можем да направим нещо по-добро.
Тъй като стойностите на `unsigned char` са в малък и фиксиран диапазон — от 0 до 255 — можем да използваме _Counting sort_.

Kазваме на компилатора:

_"Използвай общия шаблон по подразбиране, но ако типът е `unsigned char`, използвай тази конкретна специализирана реализация"_

Тоест общият шаблон описва стандартното поведение, а специализацията задава специален случай за точно определен тип.

---

## Специализация и function overloading не са едно и също

Следният код:

```cpp
template <>
void print<char>(const char* arr, size_t size) {
    // ...
}
```

е **специализация**.

Но следният код:

```cpp
void print(const char* arr, size_t size) {
    // ...
}
```

не е специализация, а **function overloading**.

Разликата е концептуална:

- при **специализация** даваме специален случай на вече съществуващ шаблон
- при **overloading** просто добавяме друга функция със същото име

И двата механизма могат да изглеждат сходно при употреба, но не са едно и също нещо.

Известен пример за специално поведение за конкретен тип е `std::vector<bool>`.

Общата идея на `std::vector<T>` е, че пази елементите като обикновен масив от обекти от тип T.
При bool обаче това не е особено ефективно, защото всяка стойност е само true или false.

Затова `std::vector<bool>` има специализирана реализация, която обикновено съхранява стойностите побитово — тоест по много булеви стойности в един байт, вместо по една стойност на байт.

И тук причината е същата като в примера с `unsigned char`: не защото bool изисква различна логика на ниво идея, а защото за този конкретен тип можем да използваме по-ефикасно вътрешно представяне

## Частична специализация

### При класовите шаблони

Класовите шаблони могат да имат **частична специализация** — специализация не за един конкретен тип, а за цял клас типове.

Пример:

```cpp
#include <iostream>

template <typename T>
class Printer {
public:
    static void print() {
        std::cout << "general type\n";
    }
};

template <typename T>
class Printer<T*> {
public:
    static void print() {
        std::cout << "pointer type\n";
    }
};

int main() {
    Printer<int>::print();    // general type
    Printer<int*>::print();   // pointer type
}
```

Тук втората версия не е за един-единствен тип, а за всички указатели.

### При функциите

Функционалните шаблони **не поддържат частична специализация**.

За функции обикновено използваме:

- пълна специализация
- или function overloading

---

## Шаблони при разделна компилация

Това е една от най-важните практически теми, свързани с шаблоните.

При обикновените класове и функции е типично да разделяме кода така:

- в `.h` файл — декларации
- в `.cpp` файл — дефиниции

При шаблоните този подход обикновено не е достатъчен.

### Причината

Компилаторът трябва да вижда **пълната дефиниция на шаблона** в момента, в който го инстанцира за конкретен тип.

Ако в header файла имаме само декларация, а дефиницията е скрита в `.cpp`, тогава друг файл, който включва само header-а, няма да разполага с достатъчно информация, за да генерира конкретната инстанция.

Резултатът често е linker грешка.

---

## Защо при шаблоните дефиницията трябва да е видима

Нека разгледаме следната идея:

```cpp
MyTemplate<int> obj(5);
```

Ако компилаторът вижда само декларацията на `MyTemplate<T>`, но не и дефиницията на неговите методи, той не може да създаде кода за `MyTemplate<int>`.

За обикновен клас това не е проблем, защото реалната дефиниция вече е компилирана отделно и linker-ът я намира.

За шаблоните обаче конкретната версия често още не съществува — тя трябва тепърва да бъде генерирана.

Именно затова дефиницията трябва да бъде достъпна там, където се използва шаблонът.

---

## Защо използваме `.hpp`

На практика шаблоните най-често се поставят в `.hpp` файл, който съдържа както декларацията, така и дефиницията.

### Пример

```cpp
// MyTemplate.hpp
#pragma once

template <typename T>
class MyTemplate {
public:
    MyTemplate(const T& obj);
    const T& get() const;

private:
    T obj;
};

template <typename T>
MyTemplate<T>::MyTemplate(const T& obj) : obj(obj) {}

template <typename T>
const T& MyTemplate<T>::get() const {
    return obj;
}
```

```cpp
// main.cpp
#include <iostream>
#include <string>
#include "MyTemplate.hpp"

int main() {
    MyTemplate<int> a(42);
    MyTemplate<std::string> b("hello");

    std::cout << a.get() << '\n';
    std::cout << b.get() << '\n';
}
```

Този подход работи, защото при `#include` компилаторът вижда пълната дефиниция и може да инстанцира шаблона.

### Важно уточнение

Разширението `.hpp` не е специално за компилатора. То е просто конвенция, с която показваме, че файлът съдържа header съдържание и имплементация на шаблони.

Може да се използва и друг подход, например `.h` + `.tpp`, стига дефиницията отново да бъде видима в момента на инстанциране.

## Експлицитно инстанциране на шаблонни класове

Съществува и друг подход: можем да държим дефинициите в `.cpp`, но тогава трябва **изрично** да кажем за кои типове искаме компилаторът да генерира код.

### Пример

```cpp
// MyTemplate.h
#pragma once
#include <string>

template <typename T>
class MyTemplate {
public:
    MyTemplate(const T& obj);
    const T& get() const;

private:
    T obj;
};
```

```cpp
// MyTemplate.cpp
#include "MyTemplate.h"

template <typename T>
MyTemplate<T>::MyTemplate(const T& obj) : obj(obj) {}

template <typename T>
const T& MyTemplate<T>::get() const {
    return obj;
}

template class MyTemplate<int>;
template class MyTemplate<std::string>;
```

Така казваме на компилатора да създаде конкретно:

- `MyTemplate<int>`
- `MyTemplate<std::string>`

### Ограничение

Ако по-късно в друг файл опитаме да използваме:

```cpp
MyTemplate<double> x(3.14);
```

ще има проблем, ако такава инстанция не е експлицитно генерирана.

Следователно този подход е приложим, но ограничава шаблона до предварително избрани типове.

Именно затова в общия случай за шаблони предпочитаме дa използваме `.hpp` файл.

## Variadic templates

Обикновеният шаблон работи с фиксиран брой шаблонни параметри:

```cpp
template <typename T>
class Box { ... };
```

или

```cpp
template <typename T, typename U>
void f(T x, U y) { ... }
```

Понякога обаче искаме да работим с **произволен брой типове** или с **произволен брой аргументи**. Точно това правят **variadic templates**.

Те използват т.нар. **parameter pack** — шаблонен параметър, който представлява не един тип, а цяла последователност от типове.

### Пример за шаблонен parameter pack

```cpp
template <typename... Types>
class Tuple;
```

Тук `Types...` означава: нула или повече типа.

Например:

- `Tuple<>`
- `Tuple<int>`
- `Tuple<int, double, char>`

всички са валидни.

### Пример за функция с произволен брой аргументи

```cpp
#include <iostream>

void print() {
    std::cout << '\n';
}

template <typename T, typename... Rest>
void print(const T& first, const Rest&... rest) {
    std::cout << first << ' ';
    print(rest...);
}

int main() {
    print(1, 2.5, "hello", 'A');
}
```

### Как работи този пример

Функцията има два случая:

- **базов случай** — `print()` без аргументи;
- **рекурсивен случай** — вземаме първия аргумент и извикваме функцията за останалите.

При извикване:

```cpp
print(1, 2.5, "hello", 'A');
```

идеята е следната:

- първо `first = 1`, а `rest... = (2.5, "hello", 'A')`
- после `first = 2.5`, а `rest... = ("hello", 'A')`
- после `first = "hello"`, а `rest... = ('A')`
- после `first = 'A'`, а `rest... = ()`
- накрая се извиква базовият случай `print()`

### Parameter pack и function parameter pack

Важно е да разграничаваме две неща:

```cpp
template <typename... Types>
```

Това е **шаблонен parameter pack** — поредица от типове.

```cpp
const Types&... args
```

Това е **function parameter pack** — поредица от параметри на функцията.

Пример:

```cpp
template <typename... Types>
void func(const Types&... args) {
    // ...
}
```

Тук:

- `Types...` е списък от типове
- `args...` е списък от аргументи

### Разгръщане на pack

Многоточието `...` се използва и за **разгръщане** на parameter pack.

Например:

```cpp
print(rest...);
```

означава: подай всички елементи на `rest` като отделни аргументи.

Друг пример:

```cpp
template <typename... Args>
void g(Args... args) {
    f(args...);
}
```

Ако `args` съдържа `(1, 2.0, 'a')`, тогава:

```cpp
f(args...);
```

се държи като:

```cpp
f(1, 2.0, 'a');
```

### Fold expressions

От C++17 нататък имаме по-удобен начин за работа с variadic templates — **fold expressions**.

Пример:

```cpp
#include <iostream>

template <typename... Args>
auto sum(const Args&... args) {
    return (args + ...);
}

int main() {
    std::cout << sum(1, 2, 3, 4) << '\n';
}
```

Тук:

```cpp
(args + ...)
```

означава „събери всички аргументи“.

За аргументите `1, 2, 3, 4` това е еквивалентно на:

```cpp
((1 + 2) + 3) + 4
```

Fold expressions често правят кода по-кратък и по-лесен за четене от рекурсивния подход.

### Кога се използват variadic templates

Variadic templates са полезни, когато:

- не знаем предварително колко аргумента ще подадем;
- реализираме контейнери или структури, подобни на `std::tuple`
- искаме да препратим произволен брой аргументи към друга функция или конструктор

## Concepts

Класическите шаблони позволяват да пишем много общ код, но често грешките при компилация стават трудни за разбиране. Ако шаблонът използва операция, която даден тип не поддържа, компилаторът може да изведе дълго и трудно за четене съобщение.

**Concepts** са механизъм от C++20, който ни позволява да описваме **какви изисквания има шаблонът към типовете**, с които работи.

С concepts кодът става:

- по-четим
- по-ясен
- по-лесен за диагностика при грешки

### Пример без concept

```cpp
template <typename T>
T add(const T& a, const T& b) {
    return a + b;
}
```

Тази функция предполага, че типът `T` поддържа `operator+`, но това не е записано явно.

### Пример с готов concept

```cpp
#include <concepts>

template <std::integral T>
T add(T a, T b) {
    return a + b;
}
```

Тук изрично казваме, че `T` трябва да е **целочислен тип**.

Валидни са например:

- `int`
- `long`
- `short`

Невалидни са например:

- `double`
- `std::string`

### Същото, написано с `requires`

```cpp
#include <concepts>

template <typename T>
requires std::integral<T>
T add(T a, T b) {
    return a + b;
}
```

Това е друг запис на същата идея.

### Собствен concept

Можем да дефинираме и свои concepts.

```cpp
#include <concepts>

template <typename T>
concept Printable = requires(T x) {
    std::cout << x;
};
```

Този concept описва типове, които могат да бъдат отпечатвани с `std::cout`.

После можем да го използваме така:

```cpp
#include <iostream>

template <Printable T>
void print(const T& value) {
    std::cout << value << '\n';
}
```

### Пример с изискване за метод

```cpp
template <typename T>
concept HasPrint = requires(T x) {
    x.print();
};

template <HasPrint T>
void printObject(const T& obj) {
    obj.print();
}
```

Тук казваме, че шаблонът може да се използва само за типове, които имат метод `print()`.

### Връзка с required functions

При обикновените шаблони често казваме, че даден тип трябва да има „необходима функция“ или „необходима операция“.

Concepts не променят тази идея — те просто ни позволяват да я **запишем явно**.

Без concept:

```cpp
template <typename T>
void f(T x) {
    x.print();
}
```

С concept:

```cpp
template <typename T>
concept HasPrint = requires(T x) {
    x.print();
};

template <HasPrint T>
void f(T x) {
    x.print();
}
```

Вторият вариант е по-ясен: още от сигнатурата на функцията се вижда какво се очаква от `T`.

### Кога са полезни concepts

Concepts са особено полезни, когато:

- искаме шаблонът да допуска само определени типове
- искаме да опишем ясно изискванията към типа

## `std::forward`

Когато пишем шаблонни функции, често искаме просто да приемем аргументите и да ги препратим към друга функция или конструктор.

На пръв поглед това изглежда лесно:

```cpp
template <typename T>
void wrapper(T&& arg) {
    target(arg);
}
```

Но тук има важен проблем: `arg` има име, а **всеки именуван параметър е lvalue израз**, независимо какъв е типът му.

Тоест ако извикаме:

```cpp
wrapper(5);
```

вътре в `wrapper`, изразът `arg` вече се държи като lvalue. Ако просто подадем `arg` нататък, губим информацията, че първоначално е бил rvalue.

### Пример с два overload-а

```cpp
#include <iostream>

void process(int& x) {
    std::cout << "lvalue\n";
}

void process(int&& x) {
    std::cout << "rvalue\n";
}
```

Сега искаме обвивка:

```cpp
template <typename T>
void wrapper(T&& arg) {
    process(arg);
}
```

Извиквания:

```cpp
int x = 5;
wrapper(x);   // lvalue
wrapper(10);  // очакваме rvalue
```

Но и в двата случая `arg` е именуван параметър и се подава като lvalue. Следователно ще се извика `process(int&)`.

### Решението: `std::forward`

```cpp
#include <iostream>
#include <utility>

void process(int& x) {
    std::cout << "lvalue\n";
}

void process(int&& x) {
    std::cout << "rvalue\n";
}

template <typename T>
void wrapper(T&& arg) {
    process(std::forward<T>(arg));
}

int main() {
    int x = 5;
    wrapper(x);   // lvalue
    wrapper(10);  // rvalue
}
```

### Каква е идеята

`std::forward<T>(arg)` запазва това дали първоначалният аргумент е бил:

- lvalue
- rvalue

Това се нарича **perfect forwarding**.

### Forwarding reference

В този контекст параметърът:

```cpp
template <typename T>
void wrapper(T&& arg)
```

се нарича **forwarding reference** (в по-стари текстове често се среща и терминът *universal reference*).

Това **не е просто rvalue reference** в обикновения смисъл. Специалното поведение идва от това, че:

- параметърът е `T&&`
- `T` се извежда от аргумента при извикване

Тогава:

- ако подадем lvalue, `T` се извежда така, че параметърът на практика става lvalue reference;
- ако подадем rvalue, параметърът остава rvalue reference.

### Кога използваме `std::forward`

Използваме `std::forward`, когато:

- пишем шаблонна функция с forwarding reference
- не искаме да променяме category-то (lvalue / rvalue) на аргумента
- просто препращаме аргумента нататък

Пример:

```cpp
#include <utility>

class Example {
public:
    Example(int, double) {}
};

template <typename... Args>
Example* createExample(Args&&... args) {
    return new Example(std::forward<Args>(args)...);
}
```

Тук:

- `Args&&...` приема произволен брой аргументи;
- `std::forward<Args>(args)...` ги препраща коректно към конструктора

### Разлика между `std::move` и `std::forward`

#### `std::move`

`std::move(x)` казва: „третирай `x` като rvalue“.

Тоест с него **винаги** превръщаме аргумента в rvalue.

#### `std::forward`

`std::forward<T>(x)` казва: „запази това, което е било първоначално“

Тоест:

- ако първоначално е било lvalue, остава lvalue;
- ако първоначално е било rvalue, остава rvalue.

### Практическо правило

- ако просто искаме да използваме move операция върху обект — използваме `std::move`
- ако пишем шаблонна обвивка и искаме да препратим аргумента така, както е дошъл — използваме `std::forward`

## Примери за шаблони в стандартната библиотека

Голяма част от STL е изградена чрез шаблони.

### Шаблонни функции

- `std::swap`
- `std::max`
- `std::min`
- `std::sort`

### Шаблонни класове и структури

- `std::vector<T>`
- `std::queue<T>`
- `std::stack<T>`
- `std::optional<T>`
- `std::pair<T1, T2>`

### Smart pointers

- `std::unique_ptr<T>`
- `std::shared_ptr<T>`
- `std::weak_ptr<T>`

Например:

```cpp
std::vector<int> numbers;
std::vector<std::string> names;
```

`std::vector` е шаблонен клас, а `std::vector<int>` и `std::vector<std::string>` са негови конкретни инстанции.
